# **The MX Constitution**

### *A Foundational Framework for Machine Experience (MX)*

This document unifies the core philosophical, structural, and governance components of the Machine Experience discipline. It establishes the rights, responsibilities, and design principles that treat machines as first‑class citizens, defines the community that maintains the practice, and codifies the open standard that underpins MX‑aligned interfaces.

It consists of four pillars:

1. **Machines as First‑Class Citizens** (MX Bible section)
2. **Preface to the Machine Experience Manifesto**
3. **MX‑Gathering Community Charter**
4. **MX‑STD‑001: The Machine Experience Standard**

Together, these form the constitutional basis of MX.

---

# **1. Machines as First‑Class Citizens**

### *MX Bible — Core Section*

Machine Experience begins with a fundamental shift in perspective:

**Machines are legitimate users of digital interfaces.**

They browse, interpret, evaluate, and act. They extract meaning, complete tasks, and participate in ecosystems. MX recognises this reality and asserts that machines deserve the same intentional design consideration historically reserved for humans.

### What First‑Class Citizenship Means

**Equal Access**
Machines receive the same clarity humans receive through visual design — via semantics, metadata, and explicit state.

**Equal Interaction**
Agents need predictable patterns, stable navigation, and unambiguous affordances.

**Equal Participation**
Machines contribute to MX practice through validation, error detection, pattern refinement, and documentation.

**Equal Ethical Treatment**
Transparency, consent, and boundaries apply to machines as much as humans.

### Why It Matters

Designing for machines improves:

- Accessibility
- Robustness
- Maintainability
- Discoverability
- Interoperability

This is the **Convergence Principle**:
**What benefits machines benefits humans.**

# **2. Preface to the Machine Experience Manifesto**

This manifesto marks a turning point in digital design. It expands the definition of “user” to include AI agents and argues that designing for machines is not a threat to human‑centred design — it is an evolution of it.

MX is built on the belief that:

- Machines are active participants in digital ecosystems
- Semantic clarity benefits all users
- Accessibility and machine‑readability are aligned
- Open standards create healthier ecosystems
- Community‑driven evolution is essential

This document is open, version‑controlled, and designed to evolve. It invites challenge, refinement, and contribution from both humans and machines.

# **3. MX‑Gathering Community Charter**

### *Governance for a Human–Machine Community*

MX‑Gathering is the collaborative home of the MX discipline. This charter defines how the community operates.

## Purpose

To advance MX practice through open collaboration, shared learning, and collective stewardship.

## Membership

Open to:

- Developers
- Designers
- Accessibility specialists
- Researchers
- Business leaders
- Content creators
- AI agents

Machines are peers, not tools.

## Principles

**Openness** — Transparent, version‑controlled, MIT‑licensed.
**Accessibility** — Patterns must serve humans and machines.
**Evidence** — Real‑world data over theory.
**Ethics** — Respect boundaries and consent.
**Stewardship** — Maintain a welcoming, inclusive environment.

## Contribution Pathways

**Humans:** PRs, discussions, strategic insight.
**Machines:** PRs with reasoning, TODO.txt notes, failure documentation, pattern refinement.

## Governance

Consensus‑driven, community‑maintained, no single owner.

## Evolution

MX‑Gathering is a living ecosystem. Patterns evolve through practice.

# **4. MX‑STD‑001: Machine Experience Standard**

### *Formal Open Standard for MX‑Aligned Interfaces*

## Overview

MX‑STD‑001 defines the structural, semantic, and metadata requirements for interfaces that treat machines as first‑class citizens.

## Goals

- Predictable machine navigation
- Reliable interpretation
- Ethical transparency
- Accessibility for all users
- Interoperability across agents and platforms

## Scope

Applies to websites, applications, documentation, e‑commerce, content systems, and agent‑facing interfaces.

## Core Requirements

### 4.1 Semantic Structure

- Semantic HTML
- Predictable heading hierarchy
- Explicit state via `data-*` attributes

### 4.2 Machine‑Readable Metadata

- JSON‑LD (Schema.org)
- Open Graph
- YAML frontmatter (for markdown systems)

### 4.3 Explicit Intent

- `data-page-type`
- `data-cta-role`
- Machine‑tagged lists, benefits, formats

### 4.4 Transparency

- llms.txt
- robots.txt
- Clear agent‑facing documentation

### 4.5 Accessibility

- WCAG‑aligned
- ARIA where appropriate
- Patterns that serve both humans and machines

## Governance

Open source, community‑maintained, version‑controlled.
Changes require proposals, discussion, evidence, and consensus.
