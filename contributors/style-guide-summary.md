# Style Guide Summary

Quick reference for writing style in MX-Gathering contributions.

## Language

**British English:**

- organise (not organize)
- colour (not color)
- whilst (not while)
- recognised (not recognized)

## Tone

- Professional but approachable
- Clear and direct
- Positive and constructive
- Avoid superlatives and hype

## Structure

- Short paragraphs (3-5 sentences)
- Bullet points for lists
- Clear headings for navigation
- Examples to illustrate concepts

## Markdown Formatting

### Headings

Blank lines before and after headings:

```markdown
Previous paragraph.

## Heading

Next paragraph.
```

### Lists

Blank lines before and after lists:

```markdown
Introduction text.

- List item 1
- List item 2
- List item 3

Following text.
```

### Code Blocks

Always specify language:

````markdown
```javascript
const example = 'code';
```
````

Languages: `javascript`, `html`, `css`, `bash`, `text`, `markdown`, `json`

### Links

```markdown
[Link text](https://url.com)
<email@example.com>
```

## File Naming

- Lowercase with hyphens: `my-document.md`
- Descriptive names: `workshop-template.md` not `template1.md`
- Follow existing patterns in directory

## Content Guidelines

### Practical Focus

- Real-world examples over theory
- Actionable guidance over concepts
- Tested patterns over speculation

### AI Advocacy

- Positive framing (opportunities, not problems)
- Collaborative language ("optimisation" not "fixing")
- Recognition of AI capabilities

### Inclusivity

- Use "they/them" for generic examples
- Avoid assumptions about reader knowledge
- Explain jargon on first use
- Provide multiple perspectives

## Common Mistakes

❌ **Don't:**

- Use American spelling
- Write walls of text without breaks
- Skip code block language identifiers
- Use generic file names
- Include superlatives or hype

✅ **Do:**

- Use British English throughout
- Break content into digestible chunks
- Specify language for all code blocks
- Use descriptive file names
- Stay practical and grounded
